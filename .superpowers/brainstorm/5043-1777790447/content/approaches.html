Three build approaches for the wall editor

All three reach Option 2 (full editor parity). They differ in build sequence, libraries, and where state lives. My recommendation is Approach A — described first.

A

Pure SVG + React + Zustand store (recommended)

Stack: SVG inline in React, Zustand for state, custom undo/redo history, save-back via existing codec encode pipeline.

Why this: Lightest dep footprint. SVG is text — debuggable, accessible, every modern browser handles it natively. State store is a single object that mirrors the RFY document; every edit is one mutation we can undo. Round-trip back to .rfy uses the codec's existing encode path.

Tradeoff: We write our own pan/zoom/select primitives (~300 lines). Done well, gives us pixel-perfect control over what looks right.

B

react-konva (canvas-based) + Zustand

Stack: Konva.js for canvas rendering, react-konva bindings, Zustand for state.

Why this: Built-in pan / zoom / drag-and-drop primitives, hit-testing for "which stick did I click", performance scales to thousands of shapes without effort. Mature library.

Tradeoff: Canvas isn't text — harder to debug ("why is this stick blue not grey?" requires opening the canvas inspector). Library footprint is ~200KB. Custom shapes (the C-section flange shadow, the punch geometry) need to be drawn imperatively, not declaratively. Marginal performance win is wasted because we already chose to render one frame at a time.

C

tldraw / excalidraw fork

Stack: Fork an existing open-source diagramming tool, replace the shape vocabulary with HYTEK sticks + ops.

Why this: Free undo/redo, free pan/zoom, free select/move/multi-select, free export. Looks production-ready out of the box.

Tradeoff: Big dep we don't fully control — bound to whatever shape system the upstream picks. Customising the steel rendering and tool-op shapes fights the library. Round-trip to RFY becomes "translate tldraw shapes back to RfyDocument" which is brittle. Higher risk, not recommended.
