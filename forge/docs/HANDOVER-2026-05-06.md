## What problem are we solving?

Replace FrameCAD Detailer's role in HYTEK's manufacturing pipeline. Specifically: take the
input XMLs HYTEK produces from their estimating system and convert them into `.rfy` files
that work in the F325iT rollformer.

## What was already tried (sessions before this one)

**Approach #1: Codec rule engine.** ~13 agents over multiple sessions built a TypeScript
rule engine that mimics Detailer's tooling logic. Reached ~80% op-level match. **Did not
work** because Detailer's logic is contextual — same XML produces different ops based on
architectural reasoning the rules can't capture (load-bearing zones, opening proximity,
header context, etc.).

**Approach #2: Reference RFY cache (Oracle cache).** For 3 captured jobs (HG260001,
HG260023, HG260044), reference RFY files from Y: drive are served bit-exact from a
content-hash-keyed cache. **Works** but only for those 3 jobs.

**Approach #3 (started today, BLOCKED): A/B test of simplifiers.** Discovered
`simplify-rp.ts` is HURTING overall match by -0.8pp. Env-var disable flags shipped.
**DEFERRED per Scott** — see `docs/simplifier-scoping-investigation.md`.

## What we're doing NOW (Approach #4: Forge)

**Architecture: Detailer is the oracle. We use Detailer programmatically to produce
authentic RFY bytes. Cache them. Serve cache hits instantly. Run Detailer in background
for cache misses. Eventually replace Detailer with a learned geometric engine derived from
the cache itself.**

Why this is different from Approach #1: we're NOT trying to mimic Detailer. We're using
Detailer as the source of truth.

Why this is different from Approach #2: cache covers EVERYTHING HYTEK runs (not just 3
specific reference jobs).

### The proven flow (works in single-shot, was the basis for the worker)

This sequence works (verified 2026-05-06 14:18 and 14:30):

1. Kill any zombie Detailer processes
2. `subprocess.Popen([detailer.exe])`
3. Wait 8 seconds for UI to settle
4. Find main window by class `TfrmContainer` (title contains "FRAMECAD Detailer v5.3.4.0
   (License valid until 28/09/2026)")
5. **Import:** `Alt+F → i → x` → `TdlgImport` opens → click "Add" button → standard
   `#32770` file picker opens → set filename via clipboard paste (typewrite breaks on
   parens in path) → click "Open" → wait → ensure plan checked via "Select All" button →
   click "Import" button → wait for `TdlgImport` to close + auto-OK any TMessageForm popups
6. **Export:** `Alt+F → e` (opens Export submenu) → press DOWN 6 times → ENTER (selects
   "Rollformer RFY file" — item 7 in the submenu) → "Export to File" dialog opens → click
   "Select All" → click "Export" button → standard `#32770` Save dialog opens → set
   filename via clipboard paste → ENTER → wait for "Export Successful" popup, dismiss
7. Verify RFY landed; rename if Detailer chose a different filename

### Critical reverse-engineered facts about Detailer 5.3.4.0

- Title bar NEVER changes on import (always shows `untitled.fcp`). Don't poll title.
- Project Tree's TTreeView descendants are NOT enumerable via win32 backend. Use
  "TdlgImport closed without error" as success signal.
- Default machine setup auto-selects from XML profile (no need to manually set combos)
- Export submenu items, in order: 3D VRML, DXF, Excel, Frame Types/Machine Setups/Steel
  Setups, 3D DXF, IFC, **Rollformer RFY file** (item 7), Upload to NEXA, FIM file
- Filename combo in file dialogs has class `ComboBoxEx32`, not `Edit` — use
  `set_focus()` + clipboard paste, NOT typewrite (autocomplete eats characters)
- Popup `TMessageForm` "Some imported sections could not be matched" is BENIGN — click OK,
  NOT Cancel. Common on first import of a job.

## Key technical facts learned today

1. **Detailer regenerates GUIDs on every export** — same XML, two Detailer runs = two
   different RFYs (different design_id, different timestamps). So bit-exact match between
   Detailer runs is impossible. Op-level match is the bar.

2. **Codec output is consistently smaller than Detailer's** — typically 50-70% the size
   because codec emits ~80% of Detailer's tooling ops + much less visualization data
   (graphics polylines, circles).

3. **HG260017 GF-LBW-70.075 numbers** (single-shot Detailer-fresh):
   - 75,552 bytes RFY
   - 1.4 MB inner XML
   - 10 frames, 170 sticks, 1,770 ops
   - Same XML run via codec rule engine: 1,603 ops (90% of Detailer's count)

4. **Aggregate gap pattern across 9 jobs** (from MULTI-JOB-COMPARE-v2.xlsx):
   - 73.1% true match (exact + ≤5mm drift)
   - Web bolt-holes: codec emits 183 vs Detailer's 2,386 (~92% missing)
   - InnerService: bimodal (over-emits on small jobs, under-emits on large)
   - Swage/Chamfer: codec OVER-emits 542+955 spurious

These confirm the rule-engine approach has fundamental issues. Forge bypasses them
entirely by using Detailer as oracle.
